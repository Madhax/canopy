# Work Model — The Domain's Work Layer Made Concrete

**Status:** Implementation-ready draft · **Date:** 2026-07-06
**Upstream:** `../domain-model.md` §3–4 (authoritative semantics), `../actuation/phase3-debts.md` (D1–D6 close here), `../manager-responsibilities.md` (adopted extensions per `README.md`).
**What this doc is:** the persistence shapes, state machines, and rules for every work-layer object the Execution Engine (`engine.md`) manipulates. Conventions follow the existing server: camelCase Pydantic boundaries, snake_case SQLite tables via `register_schema`, ids from `ids.py` (`in_`, `as_`, `gt_`, `pl_`, `st_`, `dv_`, `dr_`, `cd_`, `nt_` + nanoid), one owner-module per table family.

---

## 1. Intent

What the operator actually asks for. Addressed to exactly one node (default: org root); the provenance root for everything below it.

```sql
CREATE TABLE work_intent (
    id            TEXT PRIMARY KEY,          -- in_xxxxxxxx
    org_id        TEXT NOT NULL,
    actuation_id  TEXT NOT NULL,
    target_node   TEXT NOT NULL,
    kind          TEXT NOT NULL DEFAULT 'episodic',   -- episodic | standing
    text          TEXT NOT NULL,
    state         TEXT NOT NULL DEFAULT 'open',        -- open | completed | failed | cancelled
    root_assignment_id TEXT,                            -- set once the root Assignment is created
    cadence_id    TEXT,                                 -- non-null when generated by a Cadence
    created_by    TEXT NOT NULL DEFAULT 'operator',
    created_at    TEXT NOT NULL,
    closed_at     TEXT
);
```

An **episodic** intent closes when its root Assignment closes. A **standing** intent stays `open` indefinitely; successive episodic intents (from the operator or from Cadences) reference it via `meta.standingIntentId` — the standing intent is the org's goal, never "done." Phase-2's informal intent submission (`POST /organizations/{id}/intents`) is subsumed: the same endpoint now creates a `work_intent` plus its root Assignment.

## 2. Assignment

The central working object: who, what, what done looks like, on whose budget, under which parent.

```sql
CREATE TABLE work_assignment (
    id             TEXT PRIMARY KEY,         -- as_xxxxxxxx
    org_id         TEXT NOT NULL,
    actuation_id   TEXT NOT NULL,
    intent_id      TEXT NOT NULL,            -- provenance root (invariant 6)
    parent_id      TEXT,                     -- parent Assignment; NULL for the intent's root
    node_id        TEXT NOT NULL,            -- the assigned Agent
    issued_by      TEXT NOT NULL,            -- manager node id, or 'operator' for the root
    state          TEXT NOT NULL DEFAULT 'created',
    brief_version  INTEGER NOT NULL DEFAULT 1,
    contract_kind  TEXT NOT NULL,            -- 'artifact' | 'attestation'
    contract_type  TEXT NOT NULL,            -- e.g. 'PullRequest', 'TestReport' (open vocabulary)
    meter_id       TEXT NOT NULL,            -- assignment-bound (closes debt D1)
    priority       INTEGER NOT NULL DEFAULT 0,   -- R3: higher first; FIFO within equal priority
    deliverable_id TEXT,                     -- set at delivery
    reassigned_from TEXT,                    -- R2 provenance link
    session_ref    TEXT,                     -- runtime session id (resume handle, cli-runtime.md)
    created_at     TEXT NOT NULL, updated_at TEXT NOT NULL, closed_at TEXT
);
CREATE INDEX ix_assignment_node   ON work_assignment (actuation_id, node_id, state);
CREATE INDEX ix_assignment_intent ON work_assignment (intent_id);

CREATE TABLE work_brief (               -- versioned briefs drive rework funding
    assignment_id TEXT NOT NULL,
    version       INTEGER NOT NULL,
    text          TEXT NOT NULL,
    artifact_refs TEXT NOT NULL DEFAULT '[]',   -- JSON array of org:// refs granted with the brief
    revised_by    TEXT,                          -- who issued this version (manager/operator)
    created_at    TEXT NOT NULL,
    PRIMARY KEY (assignment_id, version)
);
```

### 2.1 Lifecycle (the domain's, verbatim, now executable)

```
created → briefed → intake ⇄ gated(clarification)
                      ↓
                   planning →(X3 checkpoint? gated(approval))→ executing → delivering → accepted → closed
                      ↑                                            |
                      └───────────────── rejected ─────────────────┘      (rework; funding rule §2.2)

any active state ⇄ gated(kind)      gating releases the node to work its queue (WIP cap §2.3)
operator hold:   any active state ⇄ paused
terminal:        cancelled | failed(reason)
```

State transitions are engine-owned; runtimes *report* progress ("intake done", "plan declared", "delivering refs") and the engine moves the row. A runtime cannot skip a state — e.g. `executing` is only entered after a funded meter exists and any plan-review checkpoint resolved. **Delegation is assignment creation** and only travels manager → direct report (invariant 4, checked against the charter's `reportNodeIds` — the same data the router already enforces channels with).

### 2.2 Rework funding (the honesty rule)

On rejection the assignment returns to `planning`. Funding follows the brief version:

- **Brief unchanged** → the same meter keeps burning. A quality failure stays the report's visible cost; a rework spiral walks into warn/hard-stop on its own.
- **Brief revised for the rework round** → the engine tops the meter up by the configured rework grant, **debited from the parent assignment's meter** (the root's parent is the intent, whose "meter" is the operator's explicit top-up approval). Re-scoping is the manager's failure and surfaces one level up.

Implementation: `brief_version` at rejection time vs. at re-entry to `planning` decides the branch; the ledger call is `raise_meter(child)` + `record` of a `transfer` SpendEvent against the parent's meter (a new `provider='canopy', model='meter-transfer'` row — rollups already group by model, so transfers are visible and filterable).

### 2.3 Queue and WIP

One `executing` assignment per node (domain rule; the directory + bus already enforce delivery-when-idle). Gated/suspended assignments release the node; **WIP cap** (default 3, per-node override in `canopy.toml`) bounds concurrent suspensions — delivery workers stop delivering new assignments to a node at cap. Gate resolution re-queues the suspended assignment at the **front** (bus priority lane: the inbox topic gains a `priority` column consumed before `seq` order; R3's manager-set priority uses the same column).

## 3. Gates

All suspension is a Gate (invariant 8). One table, five kinds, one resolution surface.

```sql
CREATE TABLE work_gate (
    id            TEXT PRIMARY KEY,          -- gt_xxxxxxxx
    assignment_id TEXT NOT NULL,
    kind          TEXT NOT NULL,             -- clarification | dependency | approval | escalation | intervention
    opened_by     TEXT NOT NULL,             -- 'system' | 'trigger:<name>' | node id | 'operator'  (X1)
    owner         TEXT NOT NULL,             -- who may resolve: node id or 'operator'
    reason        TEXT NOT NULL,             -- human-readable; for triggers, the trigger payload JSON
    payload       TEXT NOT NULL DEFAULT '{}',-- kind-specific (question text, governed action, dep list…)
    state         TEXT NOT NULL DEFAULT 'open',   -- open | resolved | expired
    resolution    TEXT,                      -- JSON: {action, note?, briefVersion?, topUp?, answer?, …}
    resolved_by   TEXT,
    created_at    TEXT NOT NULL, resolved_at TEXT
);
CREATE INDEX ix_gate_open ON work_gate (state, owner);
```

| Kind | Opened by | Owner | Resolution |
|---|---|---|---|
| **clarification** | the assigned agent at intake (feasibility check fails) | issuing manager (operator for root) | a **revised brief** (new `work_brief` version) or cancel. Closes D3: the A2A `rejected + reason` shim becomes this gate. |
| **dependency** | engine at delegation (per declared dependency) | nobody — mechanical | auto-resolves when the last upstream deliverable is **accepted**; accepted refs are appended to the downstream brief's `artifact_refs` and the brief re-versioned (revision attributed to `system`, exempt from the rework-funding rule). |
| **approval** | agent reaching a governed action, or engine at a governed transition (X3) | manager if within granted limits, else operator | approve (resume) / **deny (prohibition — agent must re-plan around it; never a rework request)**. |
| **escalation** | agent asking above its pay grade (`escalate` tool) | manager (may re-own upward) | an answer: text and/or artifact refs, injected into the session on resume. Closes D4's `input-required` shim. |
| **intervention** | platform triggers (§6) **or an authority's judgment (X1)** — `opened_by` records which | manager first, operator beyond bounds | resume / redirect (revised brief) / constrain (directive, R4) / reassign (R2) / top-up / cancel. |

Gate mechanics are uniform: opening a gate moves the assignment to `gated`, snapshots `session_ref` for resume, releases the node, and emits a notification (`engine.md` §7). Resolving re-queues front-of-line with the resolution payload delivered as the next session input.

## 4. Plan, PlanStage, Step

The plan is observable state, not chain-of-thought. Agents declare and update it via MCP tools; the engine stores it; the inspector renders it.

```sql
CREATE TABLE work_plan (
    id            TEXT PRIMARY KEY,          -- pl_xxxxxxxx
    assignment_id TEXT NOT NULL,
    version       INTEGER NOT NULL DEFAULT 1,     -- revisions are versioned and visible
    created_at    TEXT NOT NULL
);
CREATE TABLE work_plan_stage (
    plan_id     TEXT NOT NULL,
    idx         INTEGER NOT NULL,
    title       TEXT NOT NULL,
    completion  TEXT NOT NULL DEFAULT '',          -- the stage's completion signal, agent-stated
    sizing      TEXT NOT NULL DEFAULT 'medium',    -- small | medium | large (advisory — never a tripwire)
    envelope_tokens INTEGER,                        -- platform-set from role defaults (§6); NULL = uncalibrated
    state       TEXT NOT NULL DEFAULT 'pending',    -- pending | active | done | dropped
    PRIMARY KEY (plan_id, idx)
);
```

**Step** is the atomic observed unit. For the `loop` runtime a Step is one gateway call (already recorded by A1). For the `cli-claude` runtime a Step is **one assistant turn** of the wrapped session, reported by the adapter with provider-authoritative usage (`cli-runtime.md` §5); a `session_span_id` groups the turns of one session for drill-down, per the envelope doc's open question — resolved in favor of per-turn Steps to keep metering honest.

```sql
CREATE TABLE work_step (
    id            TEXT PRIMARY KEY,          -- st_xxxxxxxx
    assignment_id TEXT NOT NULL,
    stage_idx     INTEGER,                   -- plan cursor at the time; NULL pre-plan
    session_span_id TEXT,
    kind          TEXT NOT NULL DEFAULT 'production',  -- coordination | production (SC-1 metric, kept)
    input_tokens  INTEGER NOT NULL, output_tokens INTEGER NOT NULL, duration_ms INTEGER NOT NULL,
    delta_kind    TEXT NOT NULL DEFAULT 'none',
    delta_ref     TEXT,                      -- artifact ref / tool name / stage idx, per delta kind
    created_at    TEXT NOT NULL
);
CREATE INDEX ix_step_assignment ON work_step (assignment_id, created_at);
```

**Delta taxonomy (closes D6)** — a closed enum, so "no delta" is mechanically detectable:

| `delta_kind` | Meaning | `delta_ref` |
|---|---|---|
| `artifact` | produced or revised an artifact | the `org://` ref |
| `tool-effect` | a granted tool changed external/workspace state | tool/grant key |
| `progress` | plan stage advanced or plan revised | stage idx / plan version |
| `message` | communicated (delegation, escalation, reply) | message id |
| `none` | tokens burned, nothing changed | — (the stall trigger's early signal) |

Every Step still lands a SpendEvent in the existing ledger (`step_id` idempotency unchanged), so spend rolls up step → stage → assignment → subtree → intent → org exactly as the domain promises.

## 5. Deliverables, Directives, Memory

```sql
CREATE TABLE work_deliverable (
    id            TEXT PRIMARY KEY,          -- dv_xxxxxxxx
    assignment_id TEXT NOT NULL,
    kind          TEXT NOT NULL,             -- artifact | attestation
    artifact_refs TEXT NOT NULL DEFAULT '[]',
    attestation   TEXT,                      -- JSON: {claim, evidenceRefs[], signedBy, at} for attestations
    summary       TEXT NOT NULL DEFAULT '',
    accepted      INTEGER,                   -- NULL pending, 1 accepted, 0 rejected
    review_note   TEXT,
    created_at    TEXT NOT NULL, reviewed_at TEXT
);

CREATE TABLE work_directive (               -- assignment-scoped only (domain); R4: issuable mid-flight
    id            TEXT PRIMARY KEY,          -- dr_xxxxxxxx
    assignment_id TEXT NOT NULL,
    issued_by     TEXT NOT NULL,
    text          TEXT NOT NULL,
    version       INTEGER NOT NULL DEFAULT 1,
    created_at    TEXT NOT NULL
);

CREATE TABLE agent_memory (                 -- durable, platform-managed (closes D5's memory half)
    org_id      TEXT NOT NULL,
    node_id     TEXT NOT NULL,
    seq         INTEGER NOT NULL,
    entry       TEXT NOT NULL,               -- JSON: {assignmentId, intentText, outcome, summary,
                                             --        costTokens, lessons?}  — written at close
    created_at  TEXT NOT NULL,
    PRIMARY KEY (org_id, node_id, seq)
);
```

**ActionAttestation** is a first-class deliverable kind (the sales-agent case): a signed claim + evidence refs, same acceptance flow and budget accounting as artifacts. The `attest_action` MCP tool records it; for governed actions the engine verifies a resolved ApprovalGate exists for the action before accepting the attestation (consented, then evidenced — invariant 9).

**Memory rules:** keyed by `org_id + node_id` (survives re-actuation — the org's people remember; deactuation doesn't lobotomize). Written by the engine at assignment close from the deliverable summary + outcome; compacted by keeping the last N entries (default 20) plus a rolling digest entry. Injected into the session context at intake as a short "your recent work" block. Inspectable (`GET`) and resettable (`DELETE` = backfilling the position) via the API — never writable by the agent directly. Phase-2's `memory.json` workspace stub is deleted.

**Workspace per assignment (closes D5's other half):** the sandbox workspace gains `assignments/<assignmentId>/` with the familiar `brief/ work/ out/` layout, provisioned at intake and left on disk for inspection (GC per the existing `canopy sandbox gc`). `work/` no longer persists across assignments; continuity across assignments is memory + artifacts, per the domain.

## 6. Intervention triggers (MVP set) and envelopes

Static role-default envelopes ship in the catalog (`roles[].effortEnvelope: {tokensPerAssignment, minutesPerAssignment}` — coarse, honest placeholders, the calibration story replaces the *values* later without touching the shape). Triggers evaluated by the engine on every step report and on a 30 s sweep:

1. **Budget warn** — meter crosses `warnThresholdPct`. Notification + node glow; work continues. (Ledger already computes `crossedWarn`.)
2. **Budget hard-stop** — reserve refused / meter exhausted. The runtime halts at the turn boundary; the engine opens an InterventionGate (`opened_by='trigger:hard-stop'`). This upgrades phase-2's `paused`-node behavior into the real gate.
3. **Stall** — no Step for `stall_minutes` (default 10) while `executing`, or ≥ K consecutive `delta_kind='none'` steps (default 5). Opens an InterventionGate.
4. **Envelope overrun** — *post-MVP* (needs trustworthy envelopes); the columns above already record what it will read.

**X1 (judgment):** `POST /assignments/{id}/intervene` opens the same gate with `opened_by='operator'` (or a manager node id, post-MVP). Halt lands at the next turn boundary — the same place the meter check sits.

## 7. Status enum (closes D2)

Directory status becomes `provisioning | idle | engaged | gated | paused | dead`. `gated` carries `gateKind` in the heartbeat payload for the chart badge. Existing values keep their meanings; phase-3 only adds — per the debt rule, no repurposing.
